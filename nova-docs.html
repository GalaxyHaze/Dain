Getting Started

Nova

A low-level, general-purpose language designed for explicit control flow, zero-cost abstractions, and a clear compilation pipeline.

v0.1.0-dev — early preview

What is Nova?

Nova is a compiled, statically-typed language with an optional interpreted mode for fast development cycles. It exposes the full compilation pipeline as distinct CLI commands — so you always know exactly what stage your code is at.

The Pipeline

Each CLI command corresponds to a stage. Every stage includes all previous stages:

check
errors only
→
compile
IR / object
→
build
native binary
→
execute
run existing
→
run is a shortcut for build + execute. Add --interpreted to any of these commands to use the bytecode path instead of native compilation.

Quick Start

shell 
# check your source for errors
nova check main.nova

# build a native binary
nova build main.nova -o bin/app

# or just run it directly
nova run main.nova

Commands

check
Parse and type-check. Report errors, produce no output.
compile
Compile to IR or object file. No linking.
build
Compile and link to a native binary.
execute
Run an existing binary. Fails if not found.
run
Build then execute in one step.

Getting Started

Installation

Nova is currently in early development. Pre-built binaries are not yet available.

Build from Source

shell 
git clone https://github.com/GalaxyHaze/Nova
cd Nova
cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build
i
Requires C++20, CMake 3.20+, and a recent version of Clang or GCC.

Verify

shell 
nova version
Nova Programming Language
Version: 0.1.0-dev

Interpreted Mode

Every command that produces or consumes a compiled artifact accepts --interpreted. This switches the target from native binary to Nova Bytecode (.nbc).

NativeInterpreted equivalent
nova compile main.novanova compile --interpreted main.nova
nova executenova execute --interpreted
nova runnova run --interpreted
i
build always produces a native binary. There is no build --interpreted.

Project Resolution

Commands that accept an optional input file will fall back to reading novaProject.toml in the current directory when no file is specified. If neither is found, the command fails with an explicit error.

docs / cli / check
CLI Reference

nova check

Parse and type-check a source file. Reports all errors to stderr. Produces no output files.

Synopsis

shell
nova check [input] [-m MODE] [-v]

Arguments

ArgumentDescription
inputSource file to check. Optional — reads novaProject.toml if omitted.

Examples

shell
# check a specific file
nova check main.nova

# check with verbose output
nova check main.nova -v

# check entry from novaProject.toml
nova check

Output

On success, prints [ok] Check passed: <file>. On failure, all errors are printed to stderr and the exit code is non-zero. No files are written to disk.

→
Use check in your editor's save hook or CI pipeline for fast feedback without touching the build artifacts.
docs / cli / compile
CLI Reference

nova compile

Compile a source file to an IR/object file (native) or bytecode file (interpreted). Does not link.

Synopsis

shell
nova compile <input> [-o OUTPUT] [--interpreted] [-m MODE] [-I DIR] [-v]

Arguments & Flags

FlagDescription
inputSource file (.nova). Required.
-o, --outputOutput file. Defaults to a.o (native) or a.nbc (interpreted).
--interpretedCompile to Nova Bytecode (.nbc) instead of native object.
-m, --modeBuild mode. See Global Flags.
-I, --includeAdd include directory. Repeatable.

Examples

shell
# compile to native object
nova compile main.nova -o main.o

# compile to bytecode
nova compile --interpreted main.nova -o main.nbc

# release mode with include paths
nova compile main.nova -m release -I ./libs -I ./vendor
docs / cli / build
CLI Reference

nova build

Compile and link to a native binary. Always produces a native executable — there is no interpreted mode for build.

Synopsis

shell
nova build [input] [-o OUTPUT] [-m MODE] [-I DIR] [-v]

Project Resolution

When input is omitted, build reads novaProject.toml from the current directory. CLI flags take precedence over toml values.

i
The -m flag only overrides the toml mode if you explicitly pass it. The default debug does not override a toml that specifies release.

Examples

shell
# build from novaProject.toml
nova build

# build a specific file
nova build main.nova -o bin/app

# release build
nova build -m release -o bin/app
docs / cli / execute
CLI Reference

nova execute

Run an existing binary or bytecode file. Does not compile anything. Fails with an explicit error if the target is not found.

Synopsis

shell
nova execute [target] [--interpreted] [-v]

Target Resolution

When target is omitted, reads output from novaProject.toml. If no toml exists, the command fails immediately.

InvocationResolves to
nova executetoml.output
nova execute --interpretedtoml.output.nbc
nova execute bin/appbin/app

Examples

shell
# run binary from toml
nova execute

# run specific binary
nova execute bin/app

# run bytecode
nova execute --interpreted bin/app.nbc
!
execute never compiles. If the binary doesn't exist, it tells you exactly which command to run first.
docs / cli / run
CLI Reference

nova run

Build and execute in a single step. Equivalent to nova build && nova execute.

Synopsis

shell
nova run [input] [-o OUTPUT] [--interpreted] [-m MODE] [-I DIR] [-v]

Interpreted Mode

With --interpreted, run calls compile --interpreted followed by execute --interpreted. No linking occurs.

CommandEquivalent to
nova runnova build && nova execute
nova run --interpretednova compile --interpreted && nova execute --interpreted

Examples

shell
# build and run from toml
nova run

# run a specific file
nova run main.nova

# interpreted mode (fast iteration)
nova run --interpreted main.nova

# release build and run
nova run -m release
docs / cli / global flags
CLI Reference

Global Flags

These flags are available on all commands that produce or consume compilation artifacts.

  • -m, --mode <MODE> Build mode. One of debug, dev, release, fast, test. Default: debug
  • -o, --output <FILE> Output file path. Defaults vary by command.
  • -I, --include <DIR> Add a directory to the import search path. Can be repeated.
  • --interpreted Use bytecode compilation path instead of native. Not available on build.
  • --emit <FORMAT> Emit an intermediate representation: ast, ir, asm, obj, bin. Not yet implemented.
  • --target <TRIPLE> Cross-compilation target triple (e.g., x86_64-linux-gnu, aarch64-apple-macos). Not yet implemented.
  • -v, --verbose Print detailed progress information.

Build Modes

ModeDescription
debugFull debug info, no optimizations. Default.
devLight optimizations, debug info retained.
releaseFull optimizations, no debug info.
fastAggressive optimizations, may sacrifice safety checks.
testEnables test harness and assertions.
docs / novaProject.toml
Project

novaProject.toml

The project manifest file. Placed at the root of your project, it lets build, run, check, and execute work without explicit file arguments.

Example

toml
# novaProject.toml

[project]
name    = "my_project"
version = "0.1.0"
entry   = "src/main.nova"

[build]
mode    = "debug"
output  = "bin/my_project"
include = ["libs", "vendor"]

Fields

[project]

FieldTypeDescription
namestringProject name.
versionstringSemantic version string.
entrystringPath to the main source file.

[build]

FieldTypeDescription
modestringDefault build mode. Overridden by -m.
outputstringOutput binary path.
includestring[]Include directories, equivalent to repeated -I.

Precedence

CLI flags always take precedence over toml values. The toml is only read when a value is not provided on the command line.

→
Running nova with no arguments in a directory that has a novaProject.toml will automatically invoke nova build.
